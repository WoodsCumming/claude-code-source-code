## 1. 项目概览

Claude Code 是 Anthropic 官方 CLI，基于 **TypeScript + Bun + React（Ink）** 构建，约 1,884 个 TS/TSX 文件。核心能力：

- 流式对话交互（Streaming API）
- 工具执行（文件读写、Bash、搜索等 47+ 内置工具）
- 多 Agent 委托（子 Agent、Fork Agent、远程 Agent）
- MCP（Model Context Protocol）扩展
- 自定义技能（Skills）
- 多层权限控制
- 对话自动压缩（防止 context 溢出）

## 8. 流式响应解析

### 整体架构

模型返回文本的解析分为三层，每层职责清晰分离：

```
HTTP response.body（ReadableStream<Uint8Array>）
    │
    ▼  【第一层】SDK：字节流 → SSE 消息块
    │  iterSSEChunks()：按双换行切割完整 SSE 块
    │  LineDecoder：Uint8Array → 单行字符串
    │  SSEDecoder：解析 event/data 字段
    ▼
SSE 消息对象 { event: "message_start", data: '{"type":...}' }
    │
    ▼  【第二层】SDK：SSE 消息 → 类型化事件对象
    │  Stream.fromSSEResponse()：JSON.parse(sse.data)
    │  过滤 ping，抛出 error 事件
    ▼
BetaRawMessageStreamEvent { type: "message_start", message: {...} }
    │
    ▼  【第三层】项目：事件 → 内容块累积 → yield 消息
    │  for await (const part of stream)
    │  switch(part.type) 分发，字符串拼接累积
    ▼
AssistantMessage / StreamEvent（yield 给上层调用方）
```

---

### 第一层：字节流 → SSE 消息

**文件：** `@anthropic-ai/sdk/core/streaming.mjs`，`@anthropic-ai/sdk/internal/decoders/line.mjs`

#### 1.1 跨 HTTP chunk 的 SSE 块拼接

`iterSSEChunks()`（`streaming.mjs:212`）维护一个 `Uint8Array` 缓冲区，每次收到网络 chunk 就追加，然后用 `findDoubleNewlineIndex()` 搜索双换行边界：

```javascript
async function* iterSSEChunks(iterator) {
    let data = new Uint8Array()
    for await (const chunk of iterator) {
        // 追加到缓冲区
        let newData = new Uint8Array(data.length + binaryChunk.length)
        newData.set(data)
        newData.set(binaryChunk, data.length)
        data = newData
        // 找到双换行就 yield 一个完整 SSE 块
        let patternIndex
        while ((patternIndex = findDoubleNewlineIndex(data)) !== -1) {
            yield data.slice(0, patternIndex)
            data = data.slice(patternIndex)
        }
    }
}
```

`findDoubleNewlineIndex()`（`line.mjs:82`）逐字节扫描，支持三种双换行格式：

| 格式 | 字节序列 |
|------|---------|
| `\n\n` | `0x0A 0x0A` |
| `\r\r` | `0x0D 0x0D` |
| `\r\n\r\n` | `0x0D 0x0A 0x0D 0x0A` |

这保证了即使一个 SSE 消息被 TCP 分片到多个 HTTP chunk，也能被正确重组。

#### 1.2 行解码与 SSE 字段解析

每个 SSE 块经 `LineDecoder`（`line.mjs:10`）逐行解码后，送入 `SSEDecoder.decode(line)`（`streaming.mjs:241`）：

```javascript
decode(line) {
    if (!line) {
        // 空行 = 一条完整 SSE 消息，返回聚合结果
        const sse = { event: this.event, data: this.data.join('\n'), raw: this.chunks }
        this.event = null; this.data = []; this.chunks = []
        return sse
    }
    let [fieldname, _, value] = partition(line, ':')  // 按第一个冒号切割
    if (fieldname === 'event') this.event = value
    if (fieldname === 'data')  this.data.push(value)
    return null
}
```

原始文本到结构体的映射示例：

```
event: message_start                        → sse.event = "message_start"
data: {"type":"message_start","message":{}} → sse.data  = '{"type":"message_start",...}'
(空行)                                      → yield sse 对象
```

`partition(line, ':')` 只切第一个冒号，确保 data 中的 JSON 值内的冒号不被误切。

### 第二层：SSE 消息 → 类型化事件对象

**文件：** `@anthropic-ai/sdk/core/streaming.mjs:19`

`Stream.fromSSEResponse()` 内部的 async generator 负责过滤和反序列化：

```javascript
for await (const sse of _iterSSEMessages(response, controller)) {
    // 白名单匹配：只处理已知事件类型
    if (sse.event === 'message_start' ||
        sse.event === 'message_delta' ||
        sse.event === 'message_stop' ||
        sse.event === 'content_block_start' ||
        sse.event === 'content_block_delta' ||
        sse.event === 'content_block_stop') {
        yield JSON.parse(sse.data)   // 字符串 → BetaRawMessageStreamEvent

    if (sse.event === 'ping')  continue          // 心跳，忽略

```

`message_start` 的匹配就是这里的字符串相等判断——`sse.event` 来自 `SSEDecoder` 解析 `event:` 行得到的字符串值。

`Stream` 类通过 `[Symbol.asyncIterator]()` 暴露为异步迭代器（`streaming.mjs:125`），这是 `for await (const part of stream)` 能工作的基础。

### 第三层：事件对象 → 内容块累积

**文件：** `src/services/api/claude.ts:1941`

#### 3.1 为什么用原始 Stream 而非 BetaMessageStream

`claude.ts:1818` 有关键注释：

> Use raw stream instead of BetaMessageStream to avoid **O(n²) partial JSON parsing**.  
> BetaMessageStream calls `partialParse()` on every `input_json_delta`.

SDK 自带的 `BetaMessageStream` 每次 delta 都对累积的 JSON 字符串做一次 `partialParse()`，随着 input 增长产生 O(n²) 开销。项目直接使用 raw `Stream`，手动管理累积，只在 `content_block_stop` 时解析一次，复杂度为 O(n)。

#### 3.2 各事件类型的处理逻辑

```
content_block_start  →  按 block.type 初始化 contentBlocks[index]
                         tool_use/server_tool_use: input = ''（字符串）
                         text: text = ''
                         thinking: thinking = ''

content_block_delta  →  字符串拼接累积
                         input_json_delta: contentBlock.input += delta.partial_json
                         text_delta:       contentBlock.text  += delta.text
                         thinking_delta:   contentBlock.thinking += delta.thinking

content_block_stop   →  normalizeContentFromAPI() 做一次性 JSON.parse
                         构造 AssistantMessage，yield 给上层

message_start        →  记录 partialMessage、ttftMs（首字节时间）、usage
message_delta        →  更新 usage、stop_reason
message_stop         →  流结束标志
```

#### 3.3 跨 chunk 的 JSON 标签解析

以 `server_tool_use` 为例，其 `input` 字段的 JSON 可能被切割成任意片段：

```
chunk 1: partial_json = '{"file_'
chunk 2: partial_json = 'path":'
chunk 3: partial_json = '"/tmp/a","content":"hello"}'
                  ↓ 字符串拼接
contentBlock.input = '{"file_path":"/tmp/a","content":"hello"}'
                  ↓ content_block_stop 时
safeParseJSON(contentBlock.input) → { file_path: "/tmp/a", content: "hello" }
```

字符串拼接天然处理了 JSON 跨 chunk 分割，无需任何特殊解析器。

#### 3.4 流的生命周期管理
